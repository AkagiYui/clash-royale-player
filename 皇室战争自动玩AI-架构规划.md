# 皇室战争自动玩AI —— 架构规划

## 总体架构

```
屏幕/录屏输入
    ↓
[分辨率归一化 + 校正]
    ↓
[感知层] 目标检测(YOLO) + 分类(卡牌/圣水) + 规则法(血条/计时器)
    ↓
[多目标跟踪层] 跨帧身份关联 + 速度估计 + 延迟外推
    ↓
[状态反序列化] → 结构化JSON状态
    ↓
[决策层] state-conditioned policy(非VLA)
    ↓
[动作执行] 坐标+卡牌+时机 → 模拟点击/拖拽
```

核心原则:**感知与决策解耦**。感知层输出干净的结构化状态,决策层完全不碰像素。这不是VLA架构,而是更接近AlphaStar/OpenAI Five的路线——对这类规则明确、UI高度结构化的游戏,这条路线比端到端方案更省成本、更好调试。

## 阶段零:运行环境与采集/执行链路(macOS为主,Linux为辅)

游戏运行在手机上,电脑(macOS主,Linux备)只负责CV和决策,不需要考虑游戏本身的运行环境。这个前提天然规避了KataCR那套方案里"必须Linux+V4L2物理采集卡+JAX(不支持Windows/对macOS支持也不成熟)"的锁定问题,因为你走的是软件层传输(adb/DroidCast/scrcpy),不涉及系统层视频设备接口。

### 采集方案对比

| 方案 | 延迟 | 原理 | 备注 |
|---|---|---|---|
| adb screencap | 最慢 | 每次完整截图+编码+传输往返 | 不建议做实时CV输入源 |
| DroidCast | 中等 | 手机端HTTP服务推JPEG流,仍是逐帧静态图机制 | 可用但非最优 |
| scrcpy | 最低 | 手机端硬件编码H264连续视频流,电脑端硬解码 | **推荐**,brew可装,SDL2+FFmpeg,macOS/Linux原生支持 |

建议直接用scrcpy作为采集层,通过其Python封装拿解码后的帧数组喂给CV pipeline,macOS和Linux代码基本可复用。

### 动作执行:minitouch/MaaTouch 优于 scrcpy控制通道 / adb input

三种触控注入方案的层级差异:

| 方案 | 原理 | 延迟 | 特点 |
|---|---|---|---|
| `adb shell input tap` | 每次单独spawn一个adb进程执行 | 最慢(百ms级) | 简单但每次有进程启动开销 |
| scrcpy控制通道 | 常驻socket,反射调用Android `InputManager.injectInputEvent`等应用层API | 较快 | 免去进程spawn,但仍走应用层输入事件分发 |
| **minitouch/MaaTouch** | 常驻socket,直接向内核输入子系统设备节点写evdev事件(经libevdev),伪装成真实物理触摸屏上报数据 | **最快** | 绕过应用层输入分发,原生支持多点触控(每指独立trackingID),延迟最低,信号最贴近真实触控行为 |

**minitouch与MaaTouch详细对比**:

| 维度 | minitouch(原版) | MaaTouch |
|---|---|---|
| 实现语言 | C(NDK交叉编译) | 100% Java |
| 部署方式 | 需按CPU架构(armeabi-v7a/arm64-v8a/x86/x86_64)+PIE/非PIE变体匹配预编译二进制 | Java字节码通过`app_process`等方式在ART/Dalvik上运行,不需按架构匹配二进制 |
| 协议命令集 | 基础触控指令:d/m/u/c/r/w | 在此基础上新增按键(`k key d/u/o`)和文本输入(`t text`)指令,是协议超集 |
| 维护状态 | 基本停止活跃维护 | 持续维护(背靠MAA明日方舟自动化项目生态) |

对皇室战争场景,两者wire协议基本等价,选MaaTouch主要理由是部署更省心(不用维护"设备架构→对应二进制"映射表)和维护活跃度更高,而非性能有本质飞跃。是否在"贴近真实物理触控信号"层面完全等价(即底层是否同样直写内核evdev节点)未找到足够确定的实现细节,如果这点对你关键,建议实测对比。

实操注意:
- 需要先把native二进制adb push到`/data/local/tmp/`并执行,这一步仍依赖adb,但只在建连时用一次,后续每次点击/拖拽走常驻socket。
- 多数设备不需要root(shell权限即可),部分定制ROM可能因权限限制需要手动开启USB调试(安全设置)相关权限。
- 视频采集(scrcpy)和触控执行(MaaTouch)可以完全解耦成两条独立通道,各自用对自己最优的实现,不必强求合并成一条通道。

### 技术栈选型:PyTorch替代JAX

- 检测/分类模型用**PyTorch + Ultralytics YOLOv8/v11**,而非JAX。PyTorch对macOS Apple Silicon有原生MPS后端(`torch.device("mps")`),Linux上直接用CUDA,同一套代码两端只需切换device字符串,不需要维护两套实现。
- 决策层(轻量MLP/Transformer)计算量小,CPU/MPS/CUDA都能跑,同样用PyTorch即可。
- 后续如需更轻量部署,可导出ONNX,在两个平台都有支持,进一步降低平台耦合。

### MetalFX辨析:不是你需要的东西

Apple的Metal生态下三个容易混淆的技术:

| 技术 | 定位 | 是否适用 |
|---|---|---|
| MetalFX | 面向3D渲染管线的时域/空域超分辨率(类似DLSS/FSR),输入是引擎渲染出的低分辨率帧,输出放大后的画面 | 不适用,你不是渲染管线 |
| MPS(Metal Performance Shaders) | 通用GPU计算库,PyTorch `mps` device后端基于此 | 已在用(见上方技术栈选型) |
| Core ML | Apple推理框架,可调度到CPU/GPU/ANE(神经网络专用引擎) | **真正值得利用的Apple专属加速** |

MetalFX是为实时3D渲染设计的上采样技术,不适合"接收视频帧+跑目标检测"这类场景,不建议尝试套用。真正该用的Apple专属加速路径是**Core ML**:Ultralytics YOLO支持导出Core ML格式,推理时系统自动调度到ANE+GPU+CPU组合,持续高频推理场景下通常比PyTorch MPS后端更快更省电。代价是需要维护两条推理部署路径(macOS走Core ML导出,Linux走CUDA/PyTorch原生),但训练代码本身不用分叉。

### macOS特有注意事项

- YOLO在MPS后端的训练性能不如同级Nvidia显卡,如需大规模训练可考虑"Linux(或云GPU)训练、macOS部署推理"分离方案。
- 部分依赖硬件编解码加速的视频处理库在macOS上的wheel支持不如Linux完善,遇到跑不动硬件解码时可退化为软件解码兜底。

## 阶段一:感知层(CV模块)

### 1.1 任务拆解,不做端到端黑盒

| 子任务 | 方法 | 备注 |
|---|---|---|
| 手牌识别(4张+下一张) | 图像分类(小型CNN/ResNet) | 卡牌不形变,模板匹配也可行 |
| 圣水条 | 规则法(像素计数) | 不需要训练模型 |
| 场上单位检测+定位+状态 | YOLOv8/v11 | 参考KataCR的9维状态标注 |
| 单位/塔血量 | 规则法(血条像素比例) | 不是OCR问题 |
| 计时器/比分 | OCR | |
| 游戏阶段 | 简单分类 | 对战中/加时/结算 |

**状态标注建议直接借鉴KataCR的设计**:每个检测目标除了类别+坐标,还应包含动作状态维度——阵营(己方/敌方)、动作(行走/攻击/部署/冻结/冲刺)、护盾或蓄力状态、可见性、暴怒、减速、治疗/复制标记,以及地面/空中、单位/建筑这类静态常量。这样很多"单位现在处于什么状态"的问题在检测这一步就直接解决了,不需要靠跨帧推断硬猜。

单位分类建议用**unit name而不是card name**去重(同一模型可能被多张卡复用)。

### 1.2 分辨率与画质适配

- **第一步做游戏区域检测+透视/仿射校正**,把任意分辨率输入统一归一化到固定虚拟画布。
- **战场坐标系用逻辑网格坐标**(如"第X列第Y行"),不用像素坐标——可以用塔的固定位置做参照锚点反推网格,不要死记网上给的数字。
- **UI元素(手牌槽/圣水条/计时器)用比例裁剪**,不用绝对像素值。
- **训练数据增强模拟画质差异**:JPEG压缩伪影、高斯模糊/噪声、亮度对比度扰动、轻微缩放抖动,给归一化误差留容忍度。

### 1.3 数据获取与标注

1. 录屏+抽帧攒几千张跨分辨率/跨画质图。
2. 少量人工标注(几百张)做种子集(用CVAT/Label Studio)。
3. 训练初版YOLO做半自动预标注,人工只修正——标注效率提升5-10倍。
4. 逐步覆盖各游戏阶段和密集团战场景。
5. 皮肤问题:训练数据必须覆盖每张卡的所有已知皮肤变体,且需要设计低成本增量微调流程(冻结骨干、只调分类头),应对皮肤持续上新。

### 1.4 可参考的现成资源

- **RoyaleAPI/cr-api-data**(GitHub):卡牌静态数据,持续同步更新(已验证包含最新卡牌),但平衡性数值改动同步有滞后,不能假设永远最新。
- **官方API**(developer.clashroyale.com):只有游戏内展示的数值,没有隐藏数值。
- **RoyaleAPI关于Hit Time/Load Time的深度解析**:揭示"Hit Speed/First Hit"是简化说法,真实机制是Hit Time+Load Time,像Sparky这类单位起手极慢,建模时要特别注意。
- **wty-yy/KataCR + 配套的Clash-Royale-Detection-Dataset/Clash-Royale-Replay-Dataset**:已跑通的完整方案,检测数据集和状态标注规范可以直接参考甚至复用,但截止2023年10月只有116种卡,需要自己补充新卡数据。

## 阶段二:跨帧关联与延迟补偿

### 2.1 是否需要跨帧身份关联

- 如果决策模型只吃"当前帧快照",不需要。
- 如果需要速度、加速度、"是否被冻结/减速"这类依赖时序差值的信息,则需要。

### 2.2 具体做法

- 皇室战争同类单位外观完全相同,**外观Re-ID没用**,应该用**基于运动的最近邻匹配**(SORT思路:卡尔曼滤波预测位置+匈牙利算法匹配)。
- 密集团战场景建议上**ByteTrack**(对低置信度检测框也尝试关联,抗遮挡更好)。
- 同类单位扎堆导致的身份内部错标,对决策通常无关紧要——不需要死磕关联精度。

### 2.3 延迟补偿

- 先做profiling,把pipeline拆段计时,很多时候优化速度本身就够,不必上复杂预测。
- 给每个状态打时间戳,不要假设检测结果是"当前时刻"。
- 用已知游戏物理参数(移动速度、圣水回复率)做确定性外推。
- 外推目标时间点应该是"决策完成+动作执行延迟"之和,不是仅补偿感知延迟。
- 决策频率不必等于感知频率:高频跑轻量防御反应检查,低频跑完整策略决策,避免决策震荡。

## 阶段三:决策层

- 输入:结构化JSON状态(手牌、场上单位含状态维度、圣水、塔血量、时间戳)。
- 输出:动作(选卡+坐标+时机)。
- 架构上不追求VLA标签,直接当state-conditioned policy处理,用轻量网络(MLP/Transformer变体)即可,训练效率远高于端到端视觉方案。
- 参考KataCR:如果走强化学习路线,注意区分**离线RL(从专家录像学,上限被专家数据卡住)**和**在线自博弈/试错学习(理论上限更高,但需要自建模拟器或用真实客户端跑,工程成本更大)**——根据你想要的AI水平上限做取舍。
- 卡牌数值(伤害、生命、圣水费用等)建议做成**外部可插拔配置**,不要硬编码进权重,方便平衡性调整后热更新而不必重新训练。

## 阶段四:维护与迭代

| 变化类型 | 影响范围 | 应对方式 |
|---|---|---|
| 新卡上线 | 检测器需要新增类别 | 增量标注+微调分类头 |
| 皮肤更新 | 检测器可能误判 | 训练数据覆盖所有皮肤,低成本增量微调 |
| 平衡性数值调整 | 决策层数值表 | 外部配置热更新,不影响CV模块 |
| 游戏UI改版 | ROI裁剪比例失效 | 归一化流程需要人工核实和调整 |

## 落地优先级建议(避免一开始就做全套)

1. 先做能跑通的最小版本:CV检测(哪怕规则简陋)→ 当前帧快照状态 → 简单规则/启发式决策 → 动作执行。全链路先跑通,再逐步替换每一环。
2. 测出真实的pipeline延迟数字,再决定要不要上跨帧跟踪和外推。
3. 决策层从规则/启发式开始,验证感知层稳定后再引入RL。
4. 只有实测发现密集团战关联错误率明显影响决策质量时,才从SORT升级到ByteTrack或更复杂方案。
5. 平衡性/皮肤维护流程从项目一开始就设计成"可插拔配置+增量微调",避免后期返工。

## 一个待你决定的关键分叉

在线自博弈(需要自建模拟器,工程量大但上限高) vs 离线模仿学习(用录像数据,起步快但上限被录像水平卡住) vs 纯规则/启发式(工程最简单,适合先跑通MVP)。三条路线可以分阶段递进,不必一步到位。
